# AI Native Engineering Workspace — 需求访谈记录

## 第一轮访谈

### Q1：团队规模与使用频率

| 角色 | 人数 |
|------|------|
| Boss | 1 人 |
| PM | 1 人 |
| 前端工程师 | 1 人 |
| 后端工程师 | 2 人 |
| QA | 1 人 |
| **合计** | **6 人** |

- 日常同时进行的项目：最多 **2 个**
- 使用频率：每小时都会打开，查看更新或记录问题

### Q2：最痛的点排序（Top 3）

**痛点 1：任务更新不及时 + 预计完成时间不明确**
- 状态需要人工维护，经常滞后
- 没有明确的时间预期

**痛点 2：测试人员无法及时收到通知**
- QA 不知道什么时候有新任务需要测试
- 信息传递依赖人工同步

**痛点 3：文档分散，缺乏关联**
- 补充内容需要新建独立文档，通过备注手动引用（如"详见XXX文档【年月日】问题几"）
- 期望：直接在任务中书写详细内容文档，支持跨任务引用
- 远期期望：知识图谱知识库 + RAG 对话分析 + AI 创建任务

### Q3：MVP 北极星指标

> MVP 第一周目标：**每个角色都大概明白系统的功能并且用起来，觉得实用而不是麻烦。**

核心要求：**实用性优先**，不能增加负担。

## 第二轮访谈

### Q4：Boss 角色的使用场景

- **主要场景**：查看项目整体进展、了解谁在做什么
- **次要场景**：可能会记录一些文档
- **不会**：亲自提 Bug 或需求
- **需求**：以"看"为主，信息浏览型用户，不需要复杂操作

### Q5：任务详细内容文档 — 典型场景

**核心问题**：目前用线上 Excel 登记任务，遇到需要详细描述的情况（复现步骤、截图、需求描述等），Excel 单元格放不下，只能新建 Word 文档，导致：
- 任务和详细描述分离，需要来回切换
- 开发人员需要逐一核对多个文档

**期望**：在任务（Issue）内直接写丰富的详细内容（Markdown + 截图 + 附件），一个 Issue 就是一个完整的上下文容器。

**最频繁的场景**：Bug 修复流程
> PM 发现 Bug → 创建 Issue → 写复现步骤 + 截图 → 指派给后端 → 后端修复后写修复说明 → 转 QA → QA 测试通过附测试截图 → 关闭

### Q6：通知方式

| 优先级 | 方式 | 说明 |
|--------|------|------|
| **P0（MVP 必须）** | 浏览器内通知（Notification API） | 类似 AI Studio 的桌面弹窗提醒 |
| **P1** | 系统内通知中心（铃铛 🔔） | 站内消息列表 |
| **P2（可选）** | 邮件 / IM 通知 | 后续迭代 |

- 期望：用户可在设置中选择通知方式，默认开启浏览器通知

## 第三轮访谈

### Q7：Issue 工作流设计

**最终工作流（6 步）**：

```
New → Suggested → Accepted → In Progress → Ready for Test → Testing → Done
                      ↑                                     
                   (可拒绝)
```

- **New**：刚创建，未分配
- **Suggested**：PM/QA 建议分配给某人（预先指派）
- **Accepted**：被指派人确认接手（认领机制）
- **In Progress**：开发中
- **Ready for Test**：开发完成，等待测试
- **Testing**：QA 测试中
- **Done**：已完成/关闭

> 补充：如果被指派人觉得不合适，可以退回 New 状态重新分配。

### Q8：项目隔离策略

- 项目之间的**文档/Issue 不隔离**，但支持**按项目分类筛选**
- 用户可以在一个界面看到所有项目的任务（跨项目视图）
- 也支持按单个项目过滤查看

### Q9：附件与文件存储

| 需求 | 方案 |
|------|------|
| 截图 | 云存储（如阿里云 OSS / Cloudflare R2） |
| PDF/Excel/Word 附件 | 云存储 |
| 单个文件大小限制 | 10MB |
| MVP 阶段 | 优先云存储，本地存储作为备选 |

### Q10：MVP 范围确认 ✅

用户确认 MVP 范围如下：

| 功能 | MVP 阶段 |
|------|:---:|
| 用户登录 + 角色权限 | ✅ |
| 项目管理（多项目） | ✅ |
| Issue CRUD（Bug/Feature/Task/Improvement） | ✅ |
| MD 描述 + 截图 + 附件 | ✅ |
| 工作流状态流转 | ✅ |
| 评论 | ✅ |
| 简单 Dashboard | ✅ |
| 浏览器通知 | ✅ |
| Git 集成 | ❌ Phase 4 |
| AI 功能 | ❌ Phase 3 |
| 知识库/RAG | ❌ Phase 3/6 |
| Sprint/Cycle 管理 | ❌ Phase 2 |
| 高级统计报表 | ❌ Phase 2 |
| 邮件/IM 通知 | ❌ Phase 2 |

**MVP 核心定位**：**用户 + 项目 + Issue（含工作流 + Markdown + 评论 + 附件）+ 简单 Dashboard + 浏览器通知**

### Q11：Assign 认领机制

- Issue 创建时，PM/QA 可以**建议分配**（Suggested 状态）
- 被建议的研发人员会收到浏览器通知
- 研发人员可以**确认接受**（→ Accepted）或**退回**（→ New）
- 这解决"不清楚该分配给谁"的情况，让研发人员自主认领
